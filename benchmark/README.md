# Co-folding training-similarity ("surprise") viewer

A web viewer that, for each weekly co-folding benchmark system (a prospective
CAMEO target run through **AlphaFold3**), shows the crystal pocket and the
predicted ligand poses **side-by-side with the single most-similar pre-cutoff
training complex** found in the PDB. The purpose is to reason about *surprise*:
did the model place a ligand it had effectively already seen (a near-identical
pre-2021 complex), or is the pose genuinely novel?

For each system the viewer can overlay:

- the **crystal** pocket (pristine RCSB structure) + crystal ligand (green, truth),
- the **AlphaFold3 predicted poses** (blue = correct <2 Å, red = wrong >=2 Å),
- the **closest pre-cutoff training ligand** (purple) and its protein,
- a **pharmacophore "cloud"** — every drug-like ligand bound in that pocket
  across Foldseek homologs, superposed in, so density = how much chemical matter
  the model has seen there.

It also reports three training-similarity metrics per system:
`train_shape_overlap` (crystal↔training), `train_pred_overlap`
(prediction↔training), and `train_memo_gap` (pred − crystal; a large positive
gap flags the prediction hugging the training ligand more than the crystal does
— a memorisation signal).

## What has a neighbour, what doesn't

Of the **47** systems in the current manifest, **43 have a most-similar
pre-cutoff training complex** (`train_pdb` set) and **4 do not**
(`9S1B`, `9T23`, `9ZL9`, `9RYB` — no pre-cutoff drug-like ligand in a
homologous pocket, i.e. genuinely novel ligands). Systems span the CAMEO weeks
2026.05.16 – 2026.06.06.

The **static demo** further applies the quiz's **≥15 heavy-atom** drug-like floor,
leaving **32** systems (drops 11 small ligands such as ASN/GLU/NIO/ETE).

## Layout

```
app/                     The viewer to host (no bulk data).
  index.html             UI; loads Mol* 4.6 from CDN. Serve app/ as web root.
  app.js                 Single-system view: crystal + poses + training overlays.
  group.js               Within-target (fragment-screen) overlay mode.
  styles.css             (styles are actually inline in index.html; kept for reference)
  systems.json           The data MANIFEST the viewer reads (schema below).
  group_a2a.json          Manifest for the A2A within-target overlay group.
  systems/               NOT committed — regenerated by prep/ (see .gitignore).

demo/                    Self-contained STATIC demo (committed, ~98 MB).
  index.html, app.js     Same viewer, paths rewritten to relative; group mode
                         stripped. Serve demo/ as web root.
  systems.json           Trimmed to the 32 systems (neighbour + >=15 heavy).
  systems/<id>/...       Baked-in per-system + training-neighbour data.

prep/                    The data pipeline (curated; no *.bak/*.log/experiments).
  build_systems.py             assemble systems + per-system files from CAMEO dumps
  prep_poses.py                extract/align AF3 predicted ligand poses
  align_to_xtal.py             superpose predictions into the crystal frame
  compute_overlaps.py          crystal/prediction shape overlaps
  add_pred_overlap.py          add prediction↔training overlap metric
  build_training_similarity.py Foldseek nearest pre-cutoff neighbour  (needs Foldseek)
  build_ligand_cloud.py        pocket pharmacophore cloud from homolog PDBs
  surprise_analysis.py         summary "surprise"/memorisation-gap analysis
  build_group.py               within-target (A2A) overlay manifest
  compute_method.py            physics-cutoff verdict (method_fail flag)
  patch_targets.py, qa_display.py, setup_systems.py   target-selection + QA helpers
  build_static_demo.py         builds demo/ from app/systems.json + viewer data
```

## `systems.json` schema

Top level: `{ "systems": [ <system>, ... ] }`. Each **system**:

| field | meaning |
|---|---|
| `id`, `ligand`, `week`, `model`, `n_atoms` | PDB id, target HET code, CAMEO week, model (AlphaFold3), ligand heavy-atom count |
| `blurb` | one-line summary (e.g. `5/5 correct, best 0.92Å`) |
| `poses` | list of predicted poses (see below) |
| `xtal_file` | crystal mmCIF (`systems/<id>/xtal.cif`) — used when single-copy |
| `xtal_1copy_file` | single-copy pocket PDB — used for multi-copy crystals |
| `xtal_center`, `n_copies`, `display_approx` | pocket centroid, crystal copy count, "display approximate" flag |
| `mindist_min`, `method_fail` | physics-cutoff distance + verdict (untrustworthy pose flag) |
| **`train_pdb`** | **the neighbour flag** — PDB id of the closest pre-cutoff training complex, or `null` if none |
| `train_het` | training complex's ligand HET |
| `train_identity`, `train_max_protein_identity` | protein sequence identity to the neighbour / best in PDB |
| `train_align_rmsd` | protein superposition RMSD to the neighbour |
| `train_shape_overlap` | crystal↔training ligand shape overlap |
| `train_pred_overlap` | prediction↔training ligand shape overlap |
| `train_memo_gap` | `train_pred_overlap − train_shape_overlap` (memorisation gap) |
| `train_offsite` | neighbour binds a different pocket (overlap 0; training ligand hidden) |
| `train_ligand_file`, `train_protein_file` | neighbour ligand / protein PDB, aligned into this frame |
| `cloud_file`, `cloud_n`, `cloud_n_pdbs`, `cloud_hets`, `cloud_best_overlap` | pocket ligand cloud |

Each **pose** in `poses[]`:

| field | meaning |
|---|---|
| `sample` | pose index (1 = top) |
| `pose_file`, `protein_file` | predicted ligand-only PDB / prediction protein PDB (aligned to crystal frame) |
| `pose_disp_file`, `protein_disp_file` | optional display-optimised variants (preferred if present) |
| `rmsd`, `correct` | pose RMSD to crystal; `correct` = RMSD < 2 Å |
| `align_n`, `align_rmsd`, `lig_match_dist`, `pose_atoms`, `confidence` | alignment diagnostics |

**The field that marks "has a most-similar training system" is `train_pdb`**
(non-null). All the `train_*` / `cloud_*` fields are only populated when it is.

### How the viewer loads files

`app/index.html` loads `app.js`, `group.js` and Mol* (from CDN); `app.js`
`fetch`es `systems.json` and then downloads each per-system PDB/CIF referenced in
the manifest, using **web-root-absolute** paths (leading `/`). **Serve `app/` as
the web root** (e.g. `cd app && python3 -m http.server 8000`). Because Mol*
loads structures via `fetch`, a static HTTP server is required — opening the
HTML directly over `file://` will not work.

The `demo/` build rewrites those `/` paths to relative `./`, so `demo/` can be
served from its own directory:

```
cd demo && python3 -m http.server 8000   # open http://localhost:8000
```

## Regenerating the data

The `app/systems/` data and the `train_*` metrics are produced by the `prep/`
pipeline from the weekly CAMEO co-folding dumps. The nearest-neighbour search in
**`build_training_similarity.py`** requires a **local Foldseek install** (it
runs `foldseek` structure search against a pre-cutoff PDB database and caches
hits under `_fshits/` and reference structures under `_refcache/` — both
git-ignored). Roughly: `build_systems.py` → `prep_poses.py` /
`align_to_xtal.py` → `compute_overlaps.py` → `build_training_similarity.py`
(Foldseek) → `add_pred_overlap.py` → `build_ligand_cloud.py`, with
`compute_method.py` adding the physics verdict and `qa_display.py` gating the
displayed geometry. Rebuild the static demo afterwards with
`prep/build_static_demo.py`.
